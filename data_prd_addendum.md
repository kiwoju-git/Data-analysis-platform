# DataLab Studio PRD 보완서 v3.0

> 이 문서는 루트의 `data_prd.md`를 보완한다. 두 문서가 충돌하면 이 문서가 우선한다.
>
> 기준 환경: **Windows 11**, **PowerShell**, **Python 3.10.x**, **CPU-only**, **localhost 단일 사용자**
>
> 제품 원칙: 기능 수보다 **통계적 타당성, 재현성, 데이터 보호, 실패 시 안전성**을 우선한다.

## 1. 보완 목적

기존 PRD는 데이터 로드부터 EDA, 통계 검정, ML, 리포트까지 폭넓게 다루지만 다음 요구사항이 충분히 명시되지 않았다.

1. 통계 검정의 가정, 효과크기, 신뢰구간, 다중비교 보정과 검정 선택 근거
2. 결측치·이상치·전처리와 ML 평가 과정에서의 데이터 누수 방지
3. 분석 재현을 위한 원본 해시, 파이프라인, 난수 시드, 패키지 버전, 분석 로그
4. Windows/Python 3.10/CPU-only에서 설치와 실행이 가능한 현실적인 범위
5. 대용량 데이터의 메모리 보호, 작업 취소, 장애 복구, 성능 수용 기준
6. 로컬 앱이라도 필요한 업로드 검증, XSS/CSV 수식 삽입, 임의 pickle 역직렬화 방지
7. 상용·제한 라이선스 및 무거운 선택 의존성에 대한 정책
8. API 자원 모델, 오류 형식, 상태 관리 및 비동기 작업 계약
9. 검증용 기준 데이터셋, 수치 허용오차, Windows CI를 포함한 품질 게이트
10. “AI 추천”이 데이터 외부 전송 또는 근거 없는 자동 해석으로 이어지지 않도록 하는 정책

## 3. 제품 범위 재정의

### 3.1 MVP(P0)

- CSV, TSV, XLSX 업로드와 명시적인 파싱 옵션
- 원본 보존, 스키마 확인, 데이터 품질 프로파일, 가상화된 미리보기
- 수정·전처리 이력, 미리보기, Undo/Redo, 데이터셋 버전
- 기술통계, 분포, 교차표, 상관분석, 핵심 가설검정
- 검정 가정, 사용 표본 수, 제외 사유, 효과크기, 95% 신뢰구간, 보정 p-value
- 선형회귀와 이항 로지스틱 회귀의 기본 진단
- 누수 방지 scikit-learn 기반 분류·회귀 베이스라인
- 분석 결과 CSV/JSON, 재현 가능한 Python 코드, HTML 리포트
- 장시간 작업 상태, 취소 요청, 오류 복구, 로컬 프로젝트 저장/복원

### 3.2 P1

- 반복측정 ANOVA와 주요 사후검정
- 부트스트랩 신뢰구간, 검정력·표본수 계산
- 하이퍼파라미터 튜닝과 제한된 모델 설명
- PDF 리포트, 정적 이미지 내보내기
- 분석 템플릿과 파이프라인 재사용
- SSE 기반 진행 이벤트
- 안전한 앱 생성 모델 아티팩트 재로딩

### 3.3 P2 또는 별도 제품 요구

- PyCaret 기반 AutoML, LIME, 광범위한 SHAP 시각화
- 시계열, 생존분석, 혼합효과모형, 복합표본/가중치 분석
- R 코드 내보내기
- 협업, 멀티유저, 사내망 배포, 인증/RBAC, 클라우드
- 외부 또는 로컬 LLM 기반 자연어 분석
- 플러그인과 사용자 정의 Python 코드 실행

P2 기능을 구현할 때는 별도 위협 모델과 통계 검증 계획을 추가해야 한다.

### 4.1 데이터 수정 원칙

- “현재 DataFrame을 직접 덮어쓰기”가 아니라 불변 데이터셋 버전을 만든다.
- Undo/Redo는 UI 상태만이 아니라 데이터셋 버전 포인터로 복구 가능해야 한다.
- 필터링된 화면과 실제 분석 대상 행을 구분하고 분석 실행 전 사용 행 수를 표시한다.
- 샘플링은 난수 시드, 방법, 표본 수, 층화 변수를 기록한다.
- 사용자 정의 수식은 허용된 함수만 사용하는 파서로 처리하며 `eval` 또는 임의 Python 실행을 사용하지 않는다.

## 5. 통계 분석 필수 요구사항

### 5.1 분석 설계 입력

분석 실행 전 사용자는 다음 메타데이터를 확인하거나 지정해야 한다.

- 변수 역할: 종속, 독립, 그룹, 공변량, subject ID, 시간, 층화, 가중치
- 측정 수준: 연속, 순서, 명목, 이진, count, 날짜시간
- 독립/대응/반복측정 여부
- 귀무·대립가설, 단측/양측, 유의수준
- 결측치 처리 방식과 분석 대상 필터
- 다중비교 보정 방법
- 사전 계획 분석인지 탐색적 분석인지

앱은 데이터 타입만으로 연구 설계를 임의 확정하지 않는다.

### 5.2 공통 출력 계약

모든 추론 통계 결과에는 가능한 범위에서 다음 항목을 포함한다.

```json
{
  "analysis_id": "uuid",
  "method": "welch_t_test",
  "method_label": "Welch 독립표본 t-검정",
  "hypothesis": {
    "alternative": "two-sided",
    "alpha": 0.05
  },
  "sample": {
    "n_total": 120,
    "n_used": 114,
    "n_excluded_missing": 6,
    "groups": {"A": 58, "B": 56}
  },
  "estimate": {"name": "mean_difference", "value": 2.31},
  "confidence_interval": {"level": 0.95, "low": 0.42, "high": 4.20},
  "test": {"statistic": 2.41, "df": 108.7, "p_value": 0.0175},
  "multiplicity": {"method": "holm", "adjusted_p_value": 0.035},
  "effect_size": {"name": "hedges_g", "value": 0.45, "ci_low": 0.08, "ci_high": 0.82},
  "assumptions": [],
  "warnings": [],
  "provenance": {}
}
```

해당되지 않는 항목은 `null`로 두되 키의 의미를 바꾸지 않는다.

### 5.3 통계 정확성 요구사항

| ID | 요구사항 | 우선순위 | 수용 기준 |
| --- | --- | --- | --- |
| STAT-001 | 기술통계 구분 | P0 | 평균/SD와 중앙값/IQR을 함께 제공하고, 분모·결측 처리·표본 SD 정의를 명시한다. |
| STAT-002 | 검정 선택 근거 | P0 | 자동 추천 시 데이터 구조, 가정, 표본 크기와 대안 검정을 사람이 읽을 수 있게 제시한다. |
| STAT-003 | 가정 확인 | P0 | 독립성은 사용자 설계 확인 대상으로 표시하고, 등분산·잔차·예상도수·선형성 등 검정별 핵심 가정을 점검한다. |
| STAT-004 | 정규성 오용 방지 | P0 | Shapiro-Wilk 결과 하나만으로 모수/비모수 검정을 자동 전환하지 않는다. Q-Q plot, 표본 크기, 강건성 경고를 함께 사용한다. |
| STAT-005 | 효과크기 | P0 | p-value와 별도로 Hedges' g/Cohen's d, rank-biserial, eta/omega squared, Cramér's V, OR 등 적절한 효과크기를 제공한다. |
| STAT-006 | 신뢰구간 | P0 | 주요 추정치와 효과크기에 기본 95% CI를 제공하고 계산 방법을 기록한다. |
| STAT-007 | 다중비교 | P0 | 여러 가설을 실행할 때 Holm을 기본 제안하고 Bonferroni와 Benjamini-Hochberg를 선택할 수 있다. 원 p와 보정 p를 모두 보존한다. |
| STAT-008 | 사후검정 | P0/P1 | ANOVA 후 Tukey, Welch ANOVA 후 Games-Howell, Kruskal-Wallis 후 Dunn 등 전체 검정과 호환되는 사후검정을 사용한다. |
| STAT-009 | 소표본·희소표 | P0 | 카이제곱 예상도수 조건을 점검하고 2x2 희소표에는 Fisher exact 또는 명시적 대안을 제공한다. |
| STAT-010 | 대응 자료 | P0 | paired test는 subject/pair ID 매칭을 검증하며 불완전한 쌍의 제외 수를 보고한다. |
| STAT-011 | 반복측정 | P1 | subject ID 필수, 구형성 및 보정 방식, 불완전 반복 관측 경고를 제공한다. |

### 5.4 검정 기본 정책

- 독립 2그룹 평균 비교는 등분산을 무조건 가정하는 Student t-test보다 Welch t-test를 기본 제안한다.
- 3그룹 이상에서 등분산이 의심되면 Welch ANOVA와 Games-Howell을 제안한다.
- 비모수 검정은 “중앙값 검정”이라고 단정하지 않고 분포 차이에 대한 검정임을 설명한다.
- p-value는 유의/비유의 이분법만 표시하지 않고 추정치, CI, 효과크기와 함께 해석한다.
- 단측 검정은 분석 실행 전에 명시적으로 선택된 경우에만 허용한다.
- 반복 탐색으로 실행된 검정 수를 세션에 기록하고 다중검정 위험을 알린다.
- 결측 메커니즘을 데이터만으로 확정하지 않는다. MCAR/MAR/MNAR는 가정과 한계로 다룬다.

## 8. 로컬 저장·세션·복구

### 8.1 저장 구조

운영 기본 경로는 `%LOCALAPPDATA%\DataLabStudio\`이며 다음을 분리한다.

- `workspaces/`: 원본, 데이터셋 버전, 분석 매니페스트
- `db/`: SQLite 메타데이터
- `exports/`: 사용자가 생성한 결과
- `logs/`: 값이 제거된 구조화 로그
- `tmp/`: 작업 임시 파일

개발 모드에서는 저장 경로를 환경변수로 변경할 수 있어야 하며 저장 디렉터리는 Git에 커밋하지 않는다.

### 8.2 복구 요구사항

- 메타데이터와 파일 쓰기는 임시 파일 후 atomic rename 방식으로 완료한다.
- 앱 재시작 후 마지막 정상 워크스페이스를 열 수 있어야 한다.
- 고아 임시 파일과 만료 세션은 시작 시 또는 명시적 정리 작업에서 제거한다.
- 삭제는 즉시 영구 삭제보다 휴지통/보존 기간 정책을 우선한다.
- SQLite migration에는 버전과 롤백 또는 백업 절차가 있어야 한다.

- 취소는 best effort임을 명시하고 부분 결과를 정상 결과로 표시하지 않는다.
- 같은 요청의 재시도가 중복 아티팩트를 만들 수 있는 경우 idempotency key를 지원한다.
- API 이벤트 루프에서 pandas/scipy/sklearn의 긴 연산을 직접 실행하지 않는다.
- 전역 mutable DataFrame 또는 사용자 간 공유 세션 상태를 사용하지 않는다.

## 10. Windows·CPU-only 구현 요구사항

- 모든 공식 개발 명령은 PowerShell에서 동작해야 한다.
- `py -3.10`, `.venv\Scripts\python.exe`, `pathlib.Path`를 기준으로 문서화한다.
- POSIX 전용 `bash`, `make`, `/tmp`, `fork`, symlink를 필수 경로로 사용하지 않는다.
- multiprocessing 코드는 Windows `spawn`을 고려하고 진입점을 보호한다.
- 파일이 열려 있을 때 rename/delete가 실패하는 Windows 동작을 테스트한다.
- UTF-8, UTF-8-SIG, CP949 파일명·내용과 한글 경로를 테스트한다.
- GPU/CUDA/PyTorch는 기본 의존성이 아니다.
- CPU 작업은 동시성·메모리·시간 제한을 적용하고 시스템 전체를 점유하지 않는다.
- 설치 시 관리자 권한, Docker, WSL, Redis를 필수로 요구하지 않는다.
- 직접 의존성은 Python 3.10 Windows wheel 가용성과 라이선스를 확인하고 고정한다.
- 네이티브 시스템 라이브러리가 필요한 패키지는 대체 경로와 설치 진단을 제공하기 전까지 P0에 넣지 않는다.

- NaN, inf, 상수열, 빈 그룹, 단일 행, 완전 분리, 특이행렬, 큰 값·작은 값을 포함한다.
- 결과 스냅샷만 비교하지 않고 수치와 경고 코드를 직접 검증한다.
- 난수 기반 부트스트랩/CV는 시드 고정 테스트와 통계적 성질 테스트를 분리한다.
- 추천 엔진은 입력 조건별 추천 이유와 금지 조건을 테이블 기반 테스트로 검증한다.

### 14.2 데이터·보안 테스트

- UTF-8-SIG, CP949, 한글 파일명/경로, 잘못된 구분자, 중복 헤더, 매우 긴 셀을 테스트한다.
- 손상 XLSX, 위장 확장자, 과도한 압축, 경로 조작, XSS 문자열, CSV 수식 문자열을 테스트한다.
- 임의 pickle 업로드와 사용자 수식의 코드 실행 시도를 거부하는 테스트를 둔다.
- 로그 캡처 테스트로 원본 값과 비밀이 기록되지 않는지 확인한다.
- 네트워크 차단 상태의 핵심 E2E 테스트를 둔다.

### 14.3 품질 게이트

- Backend: pytest, ruff, mypy, 핵심 통계 모듈의 높은 분기 커버리지
- Frontend: ESLint, TypeScript strict, Vitest, 접근성 검사
- E2E: Playwright로 업로드 → 프로파일 → 검정 → 내보내기 핵심 경로
- CI: Windows runner + Python 3.10을 필수로 한다.
- 의존성 lockfile, 라이선스 목록, 취약점 점검 결과를 유지한다.
- 성능 회귀는 기준 데이터셋과 허용 범위를 코드로 관리한다.

## 15. 단계별 개발 게이트

### Gate A: 기반

- 저장소 구조, PowerShell 부트스트랩/실행/검사 스크립트
- FastAPI health, React shell, SQLite migration
- dataset/job/result 공통 스키마
- 구조화 오류·로그·설정
- Windows CI

**종료 기준:** 새 Windows 환경에서 문서화된 한 명령으로 설치하고 health/UI smoke test가 통과한다.

### Gate B: 통계 MVP

- 안전한 업로드와 스키마 확인
- 원본/버전/lineage 저장
- 프로파일과 가상화 미리보기
- 핵심 전처리
- 기술통계, t/Welch, paired, ANOVA/Welch, 비모수, 카이제곱/Fisher, 상관
- 효과크기, CI, 다중보정, 표준 결과 스키마
- HTML/CSV/JSON/Python 내보내기

**종료 기준:** 기준 데이터셋에서 독립 참조 결과와 허용오차 내 일치하고, 분석 매니페스트로 재실행이 가능하다.

### Gate C: 회귀·ML

- 선형/로지스틱 회귀 진단
- 누수 방지 Pipeline/ColumnTransformer
- 분류·회귀 베이스라인과 올바른 split/CV
- 모델 카드, 새 데이터 스키마 검증, 안전한 앱 아티팩트

**종료 기준:** 누수 방지 테스트와 holdout 재평가가 통과하며 동일 시드에서 재현된다.

### Gate D: 고급 기능

- 제한된 튜닝, SHAP, PDF, 반복측정, 검정력
- 제품 사용성·성능 개선
- 별도 승인을 받은 라이선스/선택 의존성

다음 게이트는 앞 게이트의 품질 기준을 낮춰서 시작하지 않는다.

## 16. 완료 정의(Definition of Done)

기능은 다음을 모두 충족해야 완료다.

1. 요구사항 ID와 수용 기준이 코드·테스트·PR 설명에 연결되어 있다.
2. 성공, 빈 입력, 잘못된 입력, 경계값, 취소, 자원 초과를 테스트했다.
3. 통계 기능은 가정, N, 제외, 효과크기, CI, 경고, 프로비넌스를 반환한다.
4. 사용자 데이터가 로그, 테스트 fixture, Git에 포함되지 않는다.
5. Windows/Python 3.10/CPU-only에서 공식 검사 명령이 통과한다.
6. API/OpenAPI, 사용자 문서, 변경된 매니페스트 스키마가 함께 갱신됐다.
7. 새 의존성은 필요성, Python 3.10 Windows 지원, CPU 동작, 라이선스를 검토했다.
8. 성능 또는 메모리 영향이 있는 변경은 벤치마크 결과를 남겼다.
9. 미완성 placeholder, 가짜 결과, 경고 무시, 무음 fallback이 없다.
10. 결과 해석은 데이터가 뒷받침하는 범위를 넘지 않는다.

## 18. 원문 PRD에 대한 편집 정정

- “분석 히스토리” 설명의 `이전 분산 결과`는 `이전 분석 결과`로 해석한다.
- `React 18+ / Next.js`, `Handsontable 또는 TanStack Table`, `Redis 또는 in-memory`처럼 선택지가 열린 항목은 본 문서의 결정표를 우선한다.
- `pickle/joblib 형태로 모델 저장/로드`는 앱 생성·검증된 아티팩트에만 적용한다.
- `AI 추천`은 MVP에서 통계 규칙 기반 추천을 의미하며 외부 모델 호출을 의미하지 않는다.
- 기존 로드맵의 AutoML·튜닝·SHAP P0 우선순위는 Gate C/D로 조정한다.
